# 经验记录

## 官网视觉不要过度 AI 化

- 用户纠正：Lattice Hub 官网首版视觉“有点过于 AI 化”，需要重新审视风格。
- 后续规则：服务治理/基础设施类官网应优先呈现可信、克制、工程化的信息架构；AI Native、MCP、Agent 可发现性可以作为差异化能力出现，但不应主导首屏视觉、标题和整体氛围。
- 设计落实：减少发光网格、节点网络、AI 口号和营销式表达；改用清晰架构图、组件关系、事实指标和文档入口来建立可信度。

## 官网实现必须逐条贴合 PRD

- 用户纠正：首页和 PRD 的差别很大，说明实现不能只抓“信息架构方向”，还要逐条核对视觉、交互和验收约束。
- 后续规则：接到 Open Design/PRD 交接稿时，先列出必须落地的硬约束，再实现；完成前按 PRD 验收项逐项自检，尤其关注主色、首屏结构、右侧产品展示形态、分流入口数量和轮播交互细节。
- 设计落实：Pole.IO 首页应使用冷蓝 light canvas，右侧治理拓扑保持开放画布，不把治理图再包进厚重卡片；docs 分流必须覆盖概览、principles、components、blog、reports 五个入口。

## 有截图时优先 1:1 复刻

- 用户纠正：首页效果仍差很多，要求按截图 1:1 复刻。
- 后续规则：当用户提供截图并要求复刻时，截图优先级高于文字总结和自主审美；必须按截图测量首屏比例、导航位置、字体尺度、留白、按钮、箭头、分页器和视觉图形，不再额外发挥。
- 设计落实：先把首屏默认状态做成截图中的 Agent Discovery 页面，再保留必要的链接和轮播交互。

## 官网顶栏保持轻量

- 用户纠正：顶部导航整体高度仍偏高，且“阅读文档”不需要做成按钮，本身应是超链。
- 后续规则：复刻截图时不能把所有右侧入口都强化成 CTA；只有首屏主行动适合按钮化，Header 内的文档入口应优先保持普通链接形态，顶栏高度要服务首屏视觉预算。
- 设计落实：桌面 Header 控制在 60px 左右，移动端控制在 56px 左右；文档入口不使用背景、圆角、阴影和额外 padding。

## 品牌资产优先使用用户提供版本

- 用户纠正：品牌 logo 已提供真实图片，不应继续使用临时手绘 SVG 或重画图形。
- 后续规则：当用户提供品牌资产时，立即替换占位/自绘资产；如果资产带白底但用于玻璃或透明容器，应生成透明背景版再引用。
- 设计落实：导航 glyph 使用 `public/lattice-hub-logo.png`，来源为用户提供的 Lattice Hub logo。

## 第三方品牌入口必须用准确图标

- 用户纠正：顶部 GitHub 入口左侧不应使用 `GitFork` 分叉图标，应使用 GitHub mark。
- 后续规则：第三方品牌入口不能用语义相近的通用图标替代；图标库缺失时，使用准确的品牌 mark SVG，而不是退而使用 lucide 近似图标。
- 设计落实：`SiteHeader` 内 GitHub 入口使用 `GitHubMark` SVG 组件。

## GitHub Pages 404 要核对部署模式

- 用户纠正：Pages 预览页面报 404，说明不能只以 workflow success 或 Pages status=built 作为上线完成。
- 后续规则：GitHub Pages 从 legacy source 切换到 Actions workflow 时，必须确认 `build_type=workflow` 后重新触发一次部署，并用公网 URL 的 HTTP 状态码验证首页、子路径和关键静态资源。
- 设计落实：`lattice-hub/lattice-hub.github.io` 已切换为 workflow 部署；重新跑 `deploy-pages.yml` 后验证 `/`、`/docs/`、`/lattice-hub-logo.png` 均返回 200。

## 文档首页应先讲产品能力

- 用户纠正：`/docs` 首页不应只是 fumadocs 目录组织说明，而应是产品能力的完整介绍页面。
- 后续规则：文档模块首页承担“产品能力总览”职责，先解释 Lattice Hub 解决什么问题、有哪些能力域、组件如何协作、典型数据流是什么，再给阅读路径；目录结构只能作为辅助信息出现。
- 设计落实：`content/docs/index.mdx` 标题使用“产品能力总览”，正文覆盖服务发现、治理规则、配置中心、权限审计、AI Registry、观测链路和多运行时接入。

## 文档架构图必须控制阅读密度

- 用户纠正：Mermaid 默认生成的架构图过大，单个节点占据过多屏幕空间，需要用 `fireworks-tech-graph` 重画，并整体检查所有架构图。
- 后续规则：文档里的架构图、数据流图和流程图应优先使用固定尺寸 SVG 资产，节点标签保持短句，正文承载细节说明；不要把 Mermaid 自动布局直接交给页面渲染，尤其不要让纵向流程图在文档页里无限拉长。
- 设计落实：图资产放在 `public/diagrams`，同名导出 PNG 作为自检产物；MDX 直接引用 SVG，页面只负责响应式缩放。

## MDX 图片不能假设 src 是字符串

- 用户纠正：`/docs` 页面截图显示 `product capability map` 破图，说明文档图片即使在本地构建通过，也可能在导出 HTML 中变成无效 `src`。
- 后续规则：Fumadocs/MDX 静态图片可能被编译为结构化对象；渲染原生 `img` 前必须解包 `.src`，并在完成前检查导出 HTML 与线上 HTML 不再包含 `src="[object Object]"`。
- 设计落实：`src/mdx-components.tsx` 统一处理 MDX 图片 `src`，docs 页面显式使用该组件映射，测试覆盖字符串路径和结构化 `{ src }` 输入。

## Fumadocs 主题变量不能被官网样式污染

- 用户纠正：Fumadocs 文档 light/dark 模式文字显示有问题，且文档格式和 Fumadocs 默认质感差距明显。
- 后续规则：首页视觉 token 只能作用在首页容器内，不能在 `:root` 覆盖 Fumadocs 使用的 `--radius-*`、背景、前景等全局变量；文档图片、表格和 prose 样式不要用裸 `article` 选择器全站覆盖。
- 设计落实：把 Lattice Hub 首页变量收敛到 `.site-shell`，`body` 使用 `--color-fd-background / --color-fd-foreground`，图片补充只限定在 `#nd-docs-layout` 内。

## 文档分区切换应使用 Fumadocs 原生 tabs

- 用户纠正：文档、博客、报告不应该只是左侧目录树里的普通目录，而应该像 Fumadocs 官方示例一样在搜索框下方的分区选择器里切换。
- 后续规则：多内容分区优先接入 `DocsLayout` 的 `tabs` / sidebar dropdown；当前分区下方只展示该分区的目录树，避免把 blog/reports 混进 docs 普通目录。
- 设计落实：`src/lib/docs-navigation.tsx` 维护「文档 / 博客 / 报告」tabs，并按当前 slug 过滤 `source.pageTree`。

## 文档目录应按产品阅读路径组织

- 用户纠正：文档页面左侧目录应参考“是什么 / 使用指南 / 最佳实践”的产品文档结构，而不是只按 Principles、Components、Blog、Reports 这类实现模块堆叠。
- 后续规则：文档分区内优先按用户任务和阅读路径组织目录；技术模块可以作为落地页存在，但侧栏标题要用“简介、功能特性、接入方式、服务端安装、控制台使用、灰度发布”等可理解的产品语言。
- 设计落实：文档分区使用虚拟目录树重组为「Lattice Hub 是什么 / 使用指南 / 最佳实践 / 原理细节」，博客和报告继续通过 Fumadocs tabs 独立切换。

## 首页能力介绍优先使用功能架构图

- 用户纠正：首页 “同一个控制面” 这一段不应只是横向卡片列表，应参考控制面覆盖数据面与业务服务的功能架构图。
- 后续规则：当首页解释控制面能力时，优先用“控制面能力域 + 运行时接入 + 业务服务”的结构关系表达；卡片只能作为局部能力域，不应替代架构图。
- 设计落实：Two operating lines 区块应呈现 Lattice Hub 控制面承载服务管理、流量治理、故障容错、配置治理，并向 SDK、Sidecar、K8s Controller 和业务服务下发治理能力。
