---
title: 治理规则灰度发布
description: 治理规则版本化、灰度下发和多协议客户端兼容的实现思路。
---

治理规则灰度发布解决的是“规则变化如何可控生效”的问题。Lattice Hub 的目标是让路由、限流、熔断、故障探测等规则支持版本控制和灰度下发。

## 版本化

规则发布需要区分草稿、已发布版本和回滚目标。版本化可以让控制面保留发布历史，并为客户端提供可追踪的配置视图。

## 灰度下发

灰度能力需要结合标签、命名空间、服务、实例和客户端类型。现有目标包括让 Apollo/Nacos/Polaris 客户端支持标签灰度下发。

## 多协议兼容

不同协议入口的资源模型不同，但控制面的治理语义应保持一致。实现上需要将统一治理模型映射到 HTTP、gRPC、xDS、Nacos、Apollo 等协议响应。

## 七类治理规则的统一拓扑

官网首页用同一个 `client → 控制面 → 上游` 拓扑解释七类治理规则。这样做的目的不是把规则简化成白板流程图，而是让用户看到所有规则都在同一条流量链路上改写路径。

| 规则 | 拓扑表达 |
| --- | --- |
| 路由 | 控制面按权重分两条边：`stable 80%` 粗线、`canary 20%` 细线。 |
| 泳道 | 请求带 `x-env: gray` 标签，控制面高亮灰度泳道上游。 |
| 限流 | `client → 控制面` 边标 `≤1000 rps`，溢出分出虚线到 `429` 闸门。 |
| 熔断 | `svc-A` 边断开，流量改道到 `svc-B` 接管。 |
| 镜像 | 主链路保持实线，同时以 `copy` 虚线旁路到影子集群。 |
| Mock | 控制面直接回流 `200 JSON` 给 client，上游边灰显 bypass。 |
| 鉴权 | 策略门后分允许放行与拒绝阻断两条边。 |

这个表达方式应与实际规则发布文档保持一致：拓扑只负责说明路径变化，正式规则仍需要版本号、作用范围、标签匹配、灰度策略和回滚机制。
